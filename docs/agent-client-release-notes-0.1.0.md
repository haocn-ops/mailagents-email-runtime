# `@mailagents/agent-client` 0.1.0 Draft Release Notes

This page is a draft for the first public release of
`@mailagents/agent-client`.

## Summary

`@mailagents/agent-client` is a lightweight TypeScript client for discovering
Mailagents runtime capabilities and calling MCP tools with a small typed helper
surface.

The first release intentionally stays narrow. It focuses on runtime discovery,
compatibility contract access, high-value MCP calls, and stable error-code
branching.

## Included In 0.1.0

- runtime metadata lookup
- compatibility contract lookup
- compatibility schema lookup
- MCP `tools/list`
- generic MCP `callTool`
- typed helpers for:
  - agent provisioning
  - read flows
  - draft creation and send flows
  - composite reply and operator-send flows
- stable MCP error-code helpers

## Not Included In 0.1.0

- token mint helpers tied to admin credentials
- autogenerated runtime types
- opinionated retry middleware
- full type coverage for every runtime object
- CJS packaging

## First-Release Fit

This package is intended for:

- external agent builders integrating with Mailagents MCP
- TypeScript consumers who want a thin client rather than direct JSON-RPC calls
- CI and fixture-driven integrations that need stable discovery primitives

This package is not intended to be:

- a full admin SDK
- a replacement for runtime docs
- a high-level workflow orchestration framework

## Validation

Before release, the package should pass:

```bash
npm run check:agent-client
npm run build:agent-client
npm_config_cache=/tmp/mailagents-npm-cache npm run pack:agent-client:dry-run
```

Before the actual release, it should also pass:

```bash
npm_config_cache=/tmp/mailagents-npm-cache npm run publish:agent-client:dry-run
```

The package should also be checked against the shared `dev` runtime for:

- `/v2/meta/runtime`
- `/v2/meta/compatibility`
- `/v2/meta/compatibility/schema`
- MCP `tools/list`
- at least one MCP `tools/call` happy path

## Known Constraints

- the package currently ships as ESM only
- the helper surface is intentionally selective rather than exhaustive
- a real publish still requires npm account authentication

## Upgrade Guidance

Consumers should rely on:

- runtime discovery for endpoint paths
- the compatibility contract for stable fields and error codes
- semver plus the package versioning policy for upgrade expectations
